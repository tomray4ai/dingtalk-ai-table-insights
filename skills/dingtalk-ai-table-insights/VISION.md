# dingtalk-ai-table-insights - 设计愿景

## 问题陈述

### 现状
企业用户使用钉钉 AI 表格管理各种业务数据：
- 销售团队：销售记录、客户跟进、目标追踪
- 项目团队：项目进度、资源分配、风险管理
- HR 团队：招聘进展、面试安排、入职管理
- 财务团队：预算执行、费用报销、回款追踪

### 痛点
1. **数据孤岛** - 相关数据分散在多个表格，难以全局洞察
2. **人工分析** - 需要手动导出、整理、分析，耗时耗力
3. **风险滞后** - 问题发现不及时，往往已经造成损失
4. **洞察缺失** - 有数据无洞察，难以支持决策

## 解决方案

### 核心价值
**让 AI 帮你从多个表格中发现洞察、识别风险、给出建议**

### 关键特性
1. **关键词筛选** - 专注特定业务/项目，而非被时间限制
2. **跨表格分析** - 发现单一表格无法看到的关联
3. **风险预警** - 主动识别问题，而非被动查询
4. **可执行建议** - 不仅告诉问题，还给出行动方案

## 设计原则

### 1. 专注 > 全面
- ❌ 不设计"分析所有表格"（太多噪音）
- ✅ 支持"分析销售相关表格"（专注业务线）
- ✅ 支持"分析华东项目表格"（专注项目）

**理由：** 用户通常关注特定业务/项目，而非全部数据。

### 2. 洞察 > 数据
- ❌ 不输出"销售表有 100 条记录"
- ✅ 输出"销售目标完成率 65%，本月可能不达标"

**理由：** 用户需要的是洞察，不是数据罗列。

### 3. 行动 > 分析
- ❌ 不输出"存在风险"
- ✅ 输出"今天内联系王五，确认 28k 订单进展"

**理由：** 分析的价值在于驱动行动。

### 4. 安全 > 便利
- ✅ 明确声明所需权限（只读）
- ✅ 明确声明所需依赖（python3, mcporter）
- ✅ 数据本地分析，不上传外部
- ✅ 数据抽样限制，避免大量传输

## 安全设计

### 权限最小化
```yaml
metadata:
  openclaw:
    requires:
      env:
        - DINGTALK_MCP_TOKEN  # 仅需 MCP 认证
      bins:
        - python3
        - mcporter
    primaryEnv: DINGTALK_MCP_TOKEN
```

### 数据保护
- **只读操作** - 仅读取表格数据，不修改
- **本地分析** - 所有分析在本地执行
- **无数据留存** - 分析结果不存储
- **抽样限制** - 每表最多 50 条记录

### 透明度
- 在 SKILL.md 中明确声明所有依赖
- 在 SKILL.md 中明确说明权限范围
- 在 SKILL.md 中明确告知数据处理方式

## 技术架构

```
用户请求
   ↓
关键词筛选 (get_accessible_tables)
   ↓
表格数据读取 (调用 dingtalk-ai-table skill)
   ↓
大模型分析 (analyze_with_llm)
   ↓
洞察报告输出
```

### 核心组件
1. **表格筛选器** - 按关键词匹配表格名称
2. **数据读取器** - 调用 dingtalk-ai-table 的 read_table.py（抽样）
3. **分析引擎** - 大模型四维分析（一致性、趋势、风险、建议）
4. **报告生成器** - 格式化输出（Markdown/JSON）

### 依赖关系
```
dingtalk-ai-table-insights
└── dingtalk-ai-table (核心依赖)
    └── 钉钉 AI 表格 MCP
```

## 使用场景

### 场景 1: 销售经理的周报准备
```bash
# 周一早上，分析销售数据准备周报
python3 scripts/analyze_tables.py --keyword "销售" --output weekly_sales.md
```

**价值：** 5 分钟生成深度洞察，而非 2 小时手动整理

### 场景 2: 项目总监的风险排查
```bash
# 每周五，检查项目风险
python3 scripts/analyze_tables.py --keyword "项目"
```

**价值：** 主动发现延期风险，而非等问题爆发

### 场景 3: HR 总监的招聘分析
```bash
# 月末，分析招聘进展
python3 scripts/analyze_tables.py --keyword "招聘"
```

**价值：** 识别招聘瓶颈，优化招聘策略

### 场景 4: 财务经理的预算监控
```bash
# 每月中旬，检查预算执行
python3 scripts/analyze_tables.py --keyword "预算"
```

**价值：** 提前发现超支风险，及时调整

## 设计反思

### 为什么不用"时间范围"筛选？
1. **技术限制** - 钉钉无访问记录 API
2. **实际需求** - 用户更关注"业务"而非"时间"
3. **替代方案** - 关键词筛选更灵活、更准确

### 为什么使用 dingtalk-ai-table？
1. **复用现有能力** - 避免重复造轮子
2. **统一接口** - 所有 AI 表格操作通过同一技能
3. **维护性** - 数据读取逻辑集中维护
4. **安全性** - 依赖经过验证的技能

### 为什么限制数据抽样？
1. **token 经济** - 避免大模型 token 超限
2. **性能考虑** - 加快分析速度
3. **安全设计** - 减少数据传输量
